# 용어 추출 가이드

도메인사전 스킬이 후보 용어를 수집하고 문제 패턴을 탐지하는 알고리즘 상세.

## 추출 소스별 전략

### 1. Spec/Interview 문서 (컨텍스트 모드)

대상: `spec.md`, `interview.md`, `team-reviews/domain-*.md`

추출 대상:
- **고빈도 명사**: 3회 이상 등장하는 도메인 명사 (3회 이상이면 핵심 개념일 가능성 높음)
- **굵은 표시 용어**: `**Cart**`, `**주문**` 등으로 강조된 용어
- **테이블 헤더 용어**: 표의 헤더에 자주 등장 (도메인 모델 후보)
- **코드 블록의 변수/타입명**: `User`, `OrderStatus` 같은 식별자

추출 명령어 예시:
```bash
# 고빈도 명사 추출 (대문자로 시작하는 영단어)
grep -oE '[A-Z][a-zA-Z]+' spec.md | sort | uniq -c | sort -rn | head -30

# 강조된 한국어 용어
grep -oE '\*\*[가-힣]+\*\*' spec.md | sort -u
```

### 2. 코드베이스 (코드베이스 모드)

대상: `src/`, `app/`, `lib/` 등 소스 디렉터리

추출 대상:

| 추출 | 도구 | 패턴 |
|------|------|------|
| 클래스명 | Grep | `class \w+`, `interface \w+`, `type \w+` |
| 함수/메서드명 | Grep | `function \w+`, `def \w+`, `\w+\s*\(` |
| 타입 별칭 | Grep | `type \w+ =`, `enum \w+` |
| 디렉토리명 | Glob | `src/*/` (도메인 경계 후보) |
| UI 한국어 라벨 | Grep | `"[가-힣]+"`, `'[가-힣]+'` |
| DB 테이블/컬럼명 | Read | migration 파일, schema.sql, models.py |
| API 엔드포인트 | Grep | `/api/{resource}/`, route 정의 |

추출 우선순위:
1. **공개 인터페이스 (export)** — 외부에 노출되는 식별자가 도메인 용어일 가능성 높음
2. **DB 스키마** — 테이블/컬럼명은 가장 안정적인 도메인 용어
3. **UI 라벨** — 사용자가 보는 표현 (영-한 매핑 핵심 소스)
4. **함수/메서드** — 동사 추출 (`createOrder`, `cancelOrder`)
5. **내부 변수/주석** — 일관성 확인용 (덜 중요)

### 3. 기존 문서

대상: `README.md`, `docs/*.md`, `CONTRIBUTING.md`

추출 대상:
- 용어 정의 섹션 (`## Glossary`, `## Terms`, `## 용어`)
- 도메인 모델 다이어그램의 노드 라벨
- 코드 예시의 식별자

## 문제 패턴 탐지 알고리즘

### 동의어 탐지 (같은 개념, 다른 단어)

휴리스틱:
1. **컨텍스트 유사도**: 두 용어가 같은 동사와 자주 결합 (`add to cart`, `add to basket` → cart=basket?)
2. **위치 유사도**: 같은 모듈/디렉토리에서 등장
3. **번역 유사도**: 한국어 표기가 같음 ("장바구니"로 번역되는 영단어들)

탐지 예시:
```
- cart (사용처: 12회, 모듈: src/shop/cart/)
- basket (사용처: 3회, 모듈: src/shop/cart/)
  → 동의어 의심: 모듈이 cart인데 basket이 섞여있음
  → 제안: basket → cart로 통일
```

### 이의어 탐지 (같은 단어, 다른 개념)

휴리스틱:
1. **동음이의 후보 사전**: 영어에서 흔한 다의어 (Order, Account, Process, Subject, Issue, Bond, ...)
2. **모듈 분산**: 같은 단어가 너무 다른 모듈에서 등장 (`Order` in shop/ vs sort/)
3. **타입 차이**: 같은 이름인데 시그니처가 완전히 다름

탐지 예시:
```
- Order (in src/shop/order.ts) — 주문(고객→상점)
- Order (in src/util/sort.ts) — 정렬 순서(asc/desc)
  → 이의어: 분리 권장 (Order vs SortOrder)
```

### 과부하 탐지 (한 단어가 너무 많은 의미)

휴리스틱:
1. **타입 분기**: `User.role === 'admin'`, `User.role === 'seller'` 같은 분기가 5+ 곳
2. **prop 폭증**: `User` 타입이 30+ 필드 (역할별 필드가 모두 한 타입에)
3. **null 필드 비율**: 많은 필드가 특정 역할에서만 채워지고 다른 역할에선 null

탐지 예시:
```
User 타입 분석:
- 28개 필드 중 11개는 role='customer'에서만 채워짐
- 9개는 role='admin'에서만 채워짐
- 8개는 role='seller'에서만 채워짐
→ 과부하: Customer / Admin / Seller로 분리 권장
```

### 영-한 불일치 탐지

휴리스틱:
1. **DB 영문 vs UI 한국어 불일치**: `users` 테이블, UI는 "고객" → 일관성 의심
2. **외래어 표기 흔들림**: "어카운트" / "계정" / "account" 혼재
3. **번역 누락**: i18n 파일에 영문 키만 있고 한국어 번역이 없음

탐지 예시:
```
- DB: users 테이블 (영문)
- UI: "고객 정보", "고객 목록" (한국어 "고객")
- 코드: const user = ... (영문 user)
→ 불일치: user(시스템 사용자)와 customer(고객)가 섞여있을 가능성
→ 사용자 확인 필요
```

### 약어 남용 탐지

휴리스틱:
1. **자음만 남은 약어**: `usrCfg`, `prdMgr` (모음 제거)
2. **3글자 약어 남발**: `crt`, `ord`, `pmt`
3. **컨벤션 충돌**: 같은 단어가 어디선 풀어쓰고(`order`) 어디선 줄임(`ord`)

처리:
- 정의가 모호한 약어는 풀어쓰기 권장
- 도메인 표준 약어(`SKU`, `URL`, `API`)는 허용

## 한국어 도메인 추출 특수 케이스

한국어 환경에서 자주 등장하는 패턴:

### 1. 한자어 vs 한글 표기

| 한자어 | 한글 | 처리 |
|--------|------|------|
| 주문(注文) | 주문 | 동일 — 한자 병기 권장 (정식 문서) |
| 환불(還拂) | 환불 | 동일 |
| 결제(決濟) vs 결재(決裁) | 다름! | 분리 명확화 — 결제=payment, 결재=approval |

### 2. 외래어 표기

| 표기 흔들림 | 표준 (국립국어원) |
|-------------|-------------------|
| 컨트롤러 / 콘트롤러 | 컨트롤러 |
| 어드민 / 애드민 | 어드민 (하지만 "관리자"가 더 좋음) |
| 어카운트 | 어카운트 (하지만 "계정"이 더 좋음) |

처리: 외래어보다 한국어 용어 우선 권장.

### 3. 한국어 동사의 명사화

| 동사 | 명사형 |
|------|--------|
| 주문하다 | 주문 |
| 결제하다 | 결제 |
| 배송하다 | 배송 |

영문에선 `order/Order`(동사/명사)가 자주 같지만, 한국어에선 명사형이 분리되어야 표 형식에 적합.

## 우선순위 가이드

후보가 너무 많을 때 핵심 30개 이내로 줄이는 우선순위:

1. **🔴 필수 (모두 포함)**
   - DB 테이블명에 대응하는 엔터티
   - API 엔드포인트의 리소스명
   - 화면/메뉴에 노출되는 용어

2. **🟠 권장 (애매한 것 위주로)**
   - 동의어/이의어/과부하 탐지된 용어
   - 외부 표준이 있는 용어 (FHIR, ISO 등)
   - 자주 인용되는 비즈니스 용어

3. **🟢 선택 (생략 가능)**
   - 단일 모듈에서만 쓰이는 내부 용어
   - 기술 인프라 용어 (cache, queue, worker — 도메인 아님)
   - 한 번만 등장하는 용어
