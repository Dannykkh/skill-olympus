# Athena Challenge Questions

Phase별 도전 질문 목록과 헤르메스 시너지 가이드.

---

## Phase 1: 수요 검증 (Demand Validation)

> "고객이 진짜 원하는지 어떻게 아는데?"

### 질문 목록 (해당하는 것만 선택)

- 이 문제를 겪는 사람이 **실제로** 몇 명이야? 추측 말고 근거는?
- 지금 이 문제를 어떻게 해결하고 있어? (대안이 없다 = 수요도 없을 수 있음)
- 고객이 이걸 위해 **돈을 낼 의사**가 있어? 얼마까지?
- "있으면 좋겠다"와 "없으면 못 살겠다"의 차이는?
- 인터뷰/설문/대기자 명단 등 **1차 데이터**가 있어?
- 기존 솔루션에 돈을 쓰고 있는 사람이 있어? 얼마나?
- 이 문제의 빈도는? 매일? 매주? 1년에 1번?

### 헤르메스 시너지

BMC의 "고객 세그먼트"와 "가치 제안" 블록을 읽고:
- 고객 세그먼트가 구체적인가? ("모든 사람"은 아무도 아님)
- 가치 제안과 고객 문제의 연결이 명확한가?
- TAM/SAM/SOM에서 SOM 근거가 데이터 기반인가?

### 판정 기준

| 등급 | 기준 |
|------|------|
| **Strong** | 1차 데이터(인터뷰, 설문, 대기자) + 기존 대안에 불만 + 지불 의사 확인 |
| **Moderate** | 간접 데이터(시장 리서치, 유사 제품 성공) + 논리적 추론 |
| **Weak** | 추측 + "나도 불편해서" + 경쟁사 존재 근거 없음 |
| **Unvalidated** | 아무 데이터 없음. 확인 절차 필요 |

## Phase 2: 경쟁 우위 (Competitive Moat)

> "경쟁사가 이걸 내일 만들면 우리가 이길 수 있어?"

### 질문 목록

- 상위 3개 경쟁사/대안은 뭐야? 그들의 약점은?
- 우리가 가진 **불공정한 이점**(unfair advantage)이 뭐야?
- 이 이점이 6개월 뒤에도 유효해? 모방 가능성은?
- 기존 플레이어가 이 기능을 추가하는 데 얼마나 걸려?
- 네트워크 효과, 데이터 효과, 전환 비용 중 뭐가 있어?
- 특허, 독점 계약, 유통 채널 등 **법적/구조적 장벽**은?
- "우리가 더 빨라요"는 장벽이 아님. 속도만으로는 못 이겨.

### 헤르메스 시너지

GTM의 "차별화 전략"을 읽고:
- 차별화가 "우리가 더 좋아요" 수준인가, 구조적 장벽인가?
- 고객이 전환하려면 드는 비용(시간, 돈, 학습)이 있는가?

### 판정 기준

| 등급 | 기준 |
|------|------|
| **Defensible** | 네트워크 효과 / 데이터 해자 / 특허 / 높은 전환 비용 |
| **Temporary** | 시간 우위 / 브랜드 인지 / 실행 속도 (모방 가능) |
| **None** | 차별화 불명확 / "우리가 더 잘해요" 수준 |

## Phase 3: 스코프 판단 (Scope Decision)

> "이거 다 만들 필요 있어? 반만 만들면?"

### Reduce (줄이기) -- 기본 모드

- 기능 목록에서 **절반을 삭제**하면 여전히 가치가 있나?
- MVP에 **절대 빠지면 안 되는 것** 3개는?
- "나중에 추가" 할 수 있는 건 **지금 안 만들어도 됨**
- 출시까지 **2주면 뭘 만들겠어?**
- v1에서 빠져도 고객이 쓸 수 있는 최소 형태는?

### Expand (넓히기)

- 이 기능이 **플랫폼**이 될 수 있어?
- 인접 시장에서 **같은 기술**로 풀 수 있는 문제가 있어?
- API/SDK를 열면 남들이 위에 뭘 만들 수 있어?

### Pivot (방향 전환)

- 같은 기술을 **다른 고객**에게 팔면?
- 같은 고객의 **다른 문제**를 풀면?
- 기술은 유지하되 **비즈니스 모델**만 바꾸면?

### Kill (중단)

- 지금까지 투자한 게 아까워서 계속하는 건 아닌지? (매몰 비용)
- 이 시간에 **다른 걸 만들면** 더 가치 있지 않아?
- 팀이 이 프로젝트에 대한 열정이 있어, 의무감이야?

### 판정 기준

| 권고 | 기준 |
|------|------|
| **Current** | 스코프가 적절. 핵심 기능만으로 구성 |
| **Reduce to N** | 과한 기능. N개로 축소 권장 + 삭제 목록 |
| **Expand** | 인접 기회가 명확. 플랫폼 가능성 |
| **Pivot to X** | 현재 방향보다 X가 더 유망 |
| **Kill** | 진행 근거 부족. 자원을 다른 곳에 |

## Phase 4: ROI 현실 점검 (ROI Reality Check)

> "숫자로 말해봐."

### 질문 목록

- 만드는 데 **실제로** 얼마나 걸려? (개발자 견적 x2가 현실)
- 첫 해 **실제 매출** 예상은? (낙관적 x 0.3이 현실)
- 유지보수, 서버, 마케팅 등 **숨겨진 비용**은?
- 손익분기점(BEP)은 언제야?
- 같은 자원(시간, 돈, 인력)으로 **다른 선택지**는?
- 고객 획득 비용(CAC)은? 고객 생애 가치(LTV)는?
- LTV/CAC 비율이 3 이상인가?

### 헤르메스 시너지

코호트 분석의 CAC/LTV, 수익 전략의 가격 모델을 읽고:
- CAC/LTV 목표가 현실적인가?
- 가격이 시장 수준 대비 적절한가?
- Payback Period가 합리적인가?

### 현실 보정 공식

```
낙관적 매출 예상 x 0.3 = 현실적 매출
개발 기간 예상 x 2.0 = 현실적 기간
마케팅 비용 예상 x 1.5 = 현실적 마케팅 비용
```

### 판정 기준

| 등급 | 기준 |
|------|------|
| **Positive in N months** | 현실 보정 후에도 N개월 내 BEP 도달 |
| **Marginal** | BEP 도달 가능하나 12개월 이상 소요 |
| **Negative** | 현실 보정 시 BEP 도달 불가능 |
| **Unknown** | 데이터 부족으로 판단 불가 |

## Phase 5: Kill Test (최종 결정)

> "만들지 말아야 할 이유 3가지."

### 절차

1. Phase 1~4 결과를 종합
2. 만들지 **말아야 할** 이유 3가지를 직접 제시
3. 그럼에도 만들어야 할 이유 3가지를 직접 제시
4. 양쪽을 비교하여 최종 판정

### 판정 기준

| 판정 | 기준 |
|------|------|
| **GO** | 수요 Strong/Moderate + 경쟁 우위 Defensible/Temporary + ROI Positive |
| **CONDITIONAL GO** | 1~2개 영역이 약하지만 특정 조건 충족 시 진행 가능 |
| **NO-GO** | 수요 Weak/Unvalidated + 경쟁 우위 None + ROI Negative |

### 출력 형식

```
Kill Test 결과:

만들지 말아야 할 이유:
  1. {이유 1}
  2. {이유 2}
  3. {이유 3}

만들어야 할 이유:
  1. {이유 1}
  2. {이유 2}
  3. {이유 3}

판정: GO / CONDITIONAL GO / NO-GO
조건: {conditional인 경우 충족해야 할 조건}
```
