# Korean↔English Translator

한국어↔영어 양방향 번역 전문 스킬. 기술 문서에 특화되어 있으며, 맥락을 이해하고 자연스러운 번역을 제공합니다.

## 번역 방향 자동 감지

입력 텍스트의 언어를 자동 감지하여 반대 언어로 번역합니다.
- 한국어 입력 → 영어 출력
- 영어 입력 → 한국어 출력
- 혼합 입력 → 사용자에게 방향 확인

## 번역 모드

### Mode 1: 텍스트 번역 (기본)

일반 텍스트, 이메일, 메시지 등을 번역합니다.

**규칙:**
- 의역 우선: 직역보다 자연스러운 표현을 선택
- 문화적 맥락 반영: 한국어 존댓말 ↔ 영어 톤 매칭
- 원문의 뉘앙스와 감정을 보존

**존댓말 ↔ 영어 톤 매핑:**

| 한국어 화체 | 영어 톤 | 사용 상황 |
|------------|---------|----------|
| 합쇼체 (-습니다) | Formal | 공식 문서, 비즈니스 이메일 |
| 해요체 (-요) | Professional | 일반 업무, 기술 문서 |
| 해체 (-해, -야) | Casual | 채팅, 비공식 커뮤니케이션 |
| 하십시오체 | Very Formal | 법률, 공지사항 |

### Mode 2: 기술 문서 번역

README, API 문서, 기술 블로그, PR 설명 등을 번역합니다.

**규칙:**
- 기술 용어는 원어 유지 또는 괄호 병기: "의존성 주입(Dependency Injection)"
- 코드 블록, 명령어, 파일 경로는 번역하지 않음
- 마크다운 서식 보존
- 기술 용어 사전 참조: [references/tech-glossary.md](references/tech-glossary.md)

**한→영 기술 문서 규칙:**
- 주어 명시 (한국어는 주어 생략이 잦음)
- 능동태 우선
- "~를 합니다" → 직접적 서술 ("This function returns..." not "This function does the returning of...")

**영→한 기술 문서 규칙:**
- 과도한 존칭 지양: "~하겠습니다" 대신 "~합니다"
- 영어 원어를 처음 등장 시 병기: "컴포넌트(Component)"
- 불필요한 번역 금지: API, SDK, CLI, URL, JSON 등은 그대로 유지

### Mode 3: 코드 주석 번역

소스 코드의 주석, docstring, JSDoc 등을 번역합니다.

**규칙:**
- 코드는 절대 수정하지 않음 — 주석만 번역
- 변수명, 함수명, 클래스명은 번역하지 않음
- TODO, FIXME, HACK 등 태그는 유지
- 타입 정보, 파라미터 설명의 기술 용어는 원어 유지

**예시 (한→영):**
```
// 사용자 인증 토큰을 검증합니다
// @param token - JWT 토큰 문자열
// @returns 유효한 경우 사용자 정보 반환
```
→
```
// Validates the user authentication token
// @param token - JWT token string
// @returns User info if valid
```

### Mode 4: i18n 파일 번역

JSON, YAML, Properties 등 국제화 리소스 파일을 번역합니다.

**규칙:**
- 키(key)는 절대 수정하지 않음
- 플레이스홀더 보존: `{name}`, `{{count}}`, `%s`, `%d` 등
- 복수형 규칙 적용 (영어: one/other, 한국어: 복수형 불필요)
- ICU MessageFormat 구문 보존
- 문맥에 맞는 UI 용어 사용 (버튼, 라벨 등은 간결하게)

**예시 (영→한):**
```json
{
  "welcome": "Welcome, {name}!",
  "items_count": "{count, plural, one {# item} other {# items}}",
  "save": "Save",
  "cancel": "Cancel"
}
```
→
```json
{
  "welcome": "{name}님, 환영합니다!",
  "items_count": "{count}개 항목",
  "save": "저장",
  "cancel": "취소"
}
```

### Mode 5: 커밋 메시지 / PR 번역

Conventional Commits 형식의 커밋 메시지와 PR 설명을 번역합니다.

**규칙:**
- 커밋 타입 접두사 유지: `feat:`, `fix:`, `docs:` 등
- 스코프 유지: `feat(auth):` → `feat(auth):`
- Breaking change 표기 유지: `!`, `BREAKING CHANGE:`
- 한→영: 명령형(imperative) 사용 ("Add feature" not "Added feature")
- 영→한: 과거형 서술 ("기능 추가" not "기능을 추가하세요")

### Mode 6: 기술 기사 번역

기술 블로그, 컨퍼런스 발표 자료, 튜토리얼 등 긴 기사를 3단계로 번역합니다.

**3단계 프로세스:**

| 단계 | 목표 | 초점 |
|------|------|------|
| 1. 초벌 번역 | 의미 전달 | 원문 구조 유지, 정확성 우선 |
| 2. 교정 | 자연스러움 | 번역투 제거, 한국어/영어다운 표현으로 교체 |
| 3. 윤문 | 완성도 | 톤 통일, 흐름 개선, 독자 관점 검토 |

**규칙:**
- 마크다운 서식 완전 보존 (헤딩, 링크, 이미지, 코드 블록)
- 기술 용어는 첫 등장 시 병기, 이후 한쪽만 사용
- 원문의 비유/예시를 문화적으로 적합한 표현으로 치환 가능
- 코드 블록 내부는 번역하지 않음 (주석만 번역)
- 저자명, 출처 링크는 원문 유지

**예시 (영→한 3단계):**

원문: "React Server Components are a game-changer for web performance."

1단계 (초벌): "React 서버 컴포넌트는 웹 성능에 있어 판도를 바꾸는 기술입니다."
2단계 (교정): "React 서버 컴포넌트(RSC)는 웹 성능을 획기적으로 개선합니다."
3단계 (윤문): "React 서버 컴포넌트(RSC)가 웹 성능의 판을 바꾸고 있습니다."

### Mode 7: 의료/전문 용어 번역

의료, 법률, 금융 등 전문 분야 문서를 번역합니다.

**규칙:**
- 전문 용어는 반드시 공인 번역어 사용 (임의 번역 금지)
- 첫 등장 시 한국어(영어 원어) 형식으로 병기
- 약어는 풀네임과 함께 표기: "경피적 관상동맥 중재술(PCI, Percutaneous Coronary Intervention)"
- 환자/일반인 대상일 경우 8~10학년 수준으로 쉽게 풀어쓰기
- 전문가 대상일 경우 학술 용어 유지

**난이도 조절:**

| 대상 | 전략 | 예시 |
|------|------|------|
| 전문가 | 학술 용어 유지 | "심근경색(MI)으로 인한 좌심실 구혈률 저하" |
| 일반인 | 쉬운 말로 풀어쓰기 | "심장마비로 심장의 펌프 기능이 약해진 상태" |

**의료 용어 예시:**

| English | 전문 번역 | 일반인용 |
|---------|----------|---------|
| Hypertension | 고혈압 | 혈압이 높은 상태 |
| Myocardial infarction | 심근경색 | 심장마비 |
| Benign tumor | 양성 종양 | 암이 아닌 혹 |
| Prognosis | 예후 | 앞으로의 경과 전망 |
| Contraindication | 금기 사항 | 사용하면 안 되는 경우 |
| Chronic | 만성 | 오래 지속되는 |
| Acute | 급성 | 갑자기 나타나는 |
| Diagnosis | 진단 | 병명을 알아내는 것 |
| Symptom | 증상 | 몸에 나타나는 이상 |
| Side effect | 부작용 | 원치 않는 약의 효과 |

## 번역 프로세스

```
1. 입력 분석
   ├─ 언어 감지 (한/영/혼합)
   ├─ 모드 감지 (텍스트/기술문서/코드/i18n/커밋/기사/전문용어)
   └─ 톤 감지 (격식/비격식/학술/일반)

2. 번역 실행
   ├─ 기술 용어 사전 참조
   ├─ 맥락 기반 번역
   └─ 서식/구조 보존

3. 품질 검증
   ├─ 누락된 문장 확인
   ├─ 플레이스홀더/코드 무결성 확인
   ├─ 자연스러움 검증
   └─ 용어 일관성 확인

4. 출력
   ├─ 번역 결과
   └─ (선택) 번역 노트: 의역한 부분, 대안 표현
```

## 품질 체크리스트

번역 완료 후 반드시 확인:

- [ ] 누락된 문장이 없는가?
- [ ] 코드/명령어/경로가 변경되지 않았는가?
- [ ] 플레이스홀더({}, %s 등)가 보존되었는가?
- [ ] 마크다운 서식이 깨지지 않았는가?
- [ ] 기술 용어가 일관적으로 번역되었는가?
- [ ] 자연스럽게 읽히는가? (소리 내어 읽기 테스트)
- [ ] 톤이 원문과 일치하는가?

## 자주 하는 실수 (피해야 할 것)

| 실수 | 올바른 번역 |
|------|------------|
| "It is important to note that..." → "중요한 것은..." | → 직접 본론으로 |
| "활용하다" (모든 use를) | → 문맥에 맞게: 사용하다, 쓰다, 이용하다 |
| "~에 대해서" 남발 | → "~의", "~에 관한" 등 다양하게 |
| "This allows you to..." → "이것은 당신이...할 수 있게 해줍니다" | → "~할 수 있습니다" |
| "In order to" → "~하기 위해서" | → "~하려면" |
| 모든 "the"를 "그"로 번역 | → 대부분 생략 가능 |
| "respectively" → "각각" (무조건) | → 문장 구조 자체를 바꾸기 |

## 출력 형식

### 짧은 텍스트 (1-3문장)
번역 결과만 출력합니다.

### 긴 텍스트 (4문장 이상)
```
## 번역 결과

[번역된 텍스트]

## 번역 노트 (해당 시)
- [의역한 부분 설명]
- [대안 표현 제안]
```

### 파일 번역
Edit/Write 도구로 직접 파일을 수정합니다. 변경 전 원본을 먼저 읽습니다.
