**적용 범위:** 젭마인(zephermine) → 대니즈팀(agent-team) → 아르고스(argos) → 미노스(minos) 파이프라인

## 1. 하네스 엔지니어링이란

하네스 엔지니어링(Harness Engineering)은 AI 코딩 에이전트의 **실행 환경을 설계하는 기법**이다.
프롬프트 엔지니어링이 에이전트에게 "말을 거는 것"이라면,
하네스 엔지니어링은 에이전트가 "달리는 트랙을 깔아주는 것"이다.

### 3축 모델

| 축 | 정의 | 질문 |
|---|---|---|
| **환경 설계** | 에이전트가 작동하는 런타임 환경 구성 | "어디서 달리나?" |
| **제약 조건** | 에이전트가 벗어날 수 없는 경계 | "어디까지만 달리나?" |
| **피드백 루프** | 에이전트의 출력을 검증하고 되돌리는 경로 | "제대로 달렸나?" |

### 파이프라인 vs 하네스

```
파이프라인 (한 방향):
  설계 → 구현 → 끝

하네스 (왕복):
  설계 → 구현 → 감리 → "설계랑 맞아?" → 안 맞으면 돌아감
```

**가는 길은 "만드는 것"이고, 돌아오는 길은 "맞는지 확인하는 것"이다.**
돌아오는 길이 없는 노드는 하네스의 구멍이다.

## 2. CPS 프레임워크

### Context-Problem-Solution

CPS는 컨설팅에서 사용하는 구조화된 사고 프레임워크다.
각 단계에서 합의를 확인(Gate)한 후 다음으로 넘어간다.

```
C (Context)  : "우리가 같은 그림을 보고 있지?" — 공유 전제 싱크
P (Problem)  : "고객이 뭐가 불편해?"           — 진짜 문제 정의
S (Solution) : "그러면 이렇게 풀자"             — 해결책 제시
```

**핵심은 순서와 분리다.** P를 말하기 전에 C가 깔려 있어야 하고, S를 말하기 전에 P가 합의되어야 한다.

### 역할 분리: 사업 CPS vs 구현 CPS

동일한 CPS 구조지만, 바라보는 대상이 다르다:

| | 헤르메스 — 사업 CPS | 젭마인 — 구현 CPS |
|---|---|---|
| **C** (전제) | 시장, 경쟁, 규제, 수익 모델 | 이해관계자, 에코시스템, 기술 스택, 기존 코드 |
| **P** (문제) | 고객이 돈 낼 만한 불편인가? | 기술적으로 뭐가 어려운가? |
| **S** (해법) | 비즈니스 모델, GTM, 가격 | 아키텍처, 섹션 분할, 설계 |

사업 판단은 사람이 하고(헤르메스/아테나), 구현 자동화는 파이프라인이 한다(젭마인/제우스).

## 4. 노드별 하네스 점검

각 노드에서 3가지를 확인한다: **입력이 보장되는가, 제약이 있는가, 검증이 있는가.**

| 노드 | 입력 보장 | 제약 조건 | 검증 (돌아오는 길) |
|---|---|---|---|
| **인터뷰** | 사용자 요청 + (헤르메스 산출물) | CPS 3-Gate: Gate 통과 전 다음 Phase 불가 | Gate 1,2,3 합의 확인 |
| **스펙** | interview.md (Gate 결과 포함) | Context Map + Problem Statement 필수 포함 | Step 23에서 CPS 섹션 존재 확인 |
| **섹션 분할** | spec.md + plan.md + flow-diagrams | 에코시스템 커버리지 체크: ❌ 있으면 진행 불가 | Backfill로 spec.md 역추적 |
| **구현** | sections/ (자립형 파일) | 500줄 제한, 코드 리뷰, Wave 의존성 | Master Checklist 검증 |
| **감리** | spec.md + 구현 코드 | Phase 0~5 순차 실행 | Phase 0: CPS 추적성 검증 |
| **테스트** | qa-scenarios.md + 구현 코드 | fix-until-pass 루프 | Playwright 전체 통과 |

## 5. CPS 추적성: 가는 길 + 돌아오는 길

### 가는 길 (설계 → 구현)

```
인터뷰 Gate 1          인터뷰 Gate 2
     │                      │
     ▼                      ▼
spec.md                spec.md
## Context Map         ## Problem Statement
  에코시스템 맵            P1, P2, P3
  '관련 섹션' 열           '해결 섹션' 열
     │                      │
     ▼                      ▼
sections/index.md      sections/index.md
  에코시스템 커버리지       섹션별 문제 매핑
  Backfill ↑               Backfill ↑
```

### 돌아오는 길 (감리 → 설계 검증)

```
argos Phase 0-1: Problem Statement의 P1~P3 각각에
                 Requirements에서 대응 솔루션이 있는가?
                 ✅/❌

argos Phase 0-2: 에코시스템 맵의 각 시스템이
                 sections/index.md에서 커버되는가?
                 ✅/⏭️/❌

argos Phase 0-3: Problem Statement의 '해결 섹션' 열이
                 가리키는 파일이 실제로 존재하는가?
                 ✅/⚠️/❌
```

### 하위 호환성

| 시나리오 | 처리 |
|---|---|
| 기존 포맷 인터뷰 (Gate 없음) | spec.md에 Context Map 없음 → argos Phase 0 건너뜀 |
| 기존 포맷 spec.md (CPS 없음) | section-index 에코시스템 체크 건너뜀 |
| CPS 인터뷰 + 기존 argos | Phase 0 조건 미충족 → Phase 1~5만 실행 |

## 7. 사용법

### A. 풀코스 (하나씩 직접)

```
/hermes              # 사업성 검토 → biz-strategy.md
/ceo                 # Go/No-Go 판정
/zephermine          # 설계 — CPS 인터뷰 + 산출물 생성
/agent-team          # 구현 — 섹션별 병렬 코딩
/argos               # 감리 — Phase 0(CPS) ~ Phase 5(도면)
/minos          # 테스트 — fix-until-pass
```

### B. 원클릭 자동화 (제우스)

```
/zeus                # 설계→구현→감리→테스트 전자동 (사용자 개입 0)
```

> 헤르메스/아테나는 제우스 밖. 사업 판단은 사람이, 구현 자동화는 제우스가.

### C. 실전 권장 플로우

```
# 1단계: 사업 검증 (첫 프로젝트면 권장)
/hermes → /ceo

# 2단계: 설계 (반드시)
/zephermine

# 3단계: 구현~테스트 (택 1)
/zeus                          # 전자동
/agent-team → /argos → /minos  # 수동 제어
```
